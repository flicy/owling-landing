# 未决问题与验证点

## 黑客松前必须决定

- Demo 是纯软件模拟玩偶，还是接入真实开发板 / 小智板子 / 麦克风 / 摄像头？
- 语音链路用哪种方案：浏览器录音 + 云端模型、移动端录音、还是硬件按钮触发？
- 成长隐喻选哪个：记忆叶子、共同故事树、伙伴等级、兴趣地图？
- 家长端是否出现：如果出现，是摘要面板、短信式提示，还是家庭故事卡片？
- 英语学习占比多大：是主功能，还是陪伴故事里的轻量元素？

## 产品定义仍需回答

- 玩偶到底是什么角色：朋友、宠物、同学、学习伙伴、家庭记录者，还是孩子的另一个小分身？
- 它和父母是什么关系：帮父母了解孩子，还是帮孩子向父母表达？
- 它应该主动到什么程度：每天问一次，还是根据事件触发？
- 它如何体现“共同成长”：记忆增多、性格变化、故事进展、能力升级，还是孩子看得见的仪式？
- 它是否需要眼睛、表情、灯光、动作，还是黑客松只用语音和屏幕表达？

## 技术验证点

- 语音识别延迟是否会破坏陪伴感？
- TTS 音色是否足够儿童友好？
- 记忆调用是否稳定，是否能准确引用孩子过去说过的内容？
- 如何避免模型胡编孩子的记忆？
- 负面情绪、安全内容、隐私内容如何处理？

## 商业与合规验证点

- 儿童大模型产品在目标市场的合规边界是什么？
- 国内演示时应该怎样表述，避免被理解成已可上市儿童大模型硬件？
- 如果出海，目标市场是北美、东南亚、还是华人家庭？
- 家长愿意为硬件付多少钱，是否接受订阅？
- IP 是自建、合作，还是先用中性原创角色？

## 黑客松后可延展

- 多 agent 社群：孩子的玩偶 agent 与其他孩子的玩偶 agent 互动。
- 亲子共同创作：玩偶把孩子和父母的对话变成故事绘本。
- 长期记忆系统：孩子的成长轨迹、兴趣变化、语言能力变化。
- 家庭回忆录：从儿童陪伴延展到家人记忆、老人陪伴和跨代沟通。
- 硬件生态：标准化接口、开源外壳、不同毛绒/IP/传感器组合。

## 当前建议

黑客松阶段先选择最小闭环：语音对话 + 轻量记忆 + 一种成长可视化 + 家长摘要。把多 agent 社群、订阅商业化、硬件量产和完整合规体系放在愿景层，不要让它们吞掉 demo。
